

## Plan: Admin Section with Design System, Brand Book & Reorganized Routes

### Overview
Create an `/admin` section that consolidates all technical/operational pages behind a role-aware gate. Add two new tabs inside it: a **Brand Book** and a **Design System** (generated by Mochi's "design agent" persona). Keep public-facing pages (Beeducation, Chat, Learning paths) clean and simple.

### What moves to Admin
- `/dashboard` (MasterControlPanel, ImageGenerator, MochiInterface, BeeEducationHub, GDPR settings)
- `/production` (ProductionDashboard)
- `/technical-details` (TechnicalDetails)
- System Help / Accessibility Helper (currently in HamburgerMenu)

### What stays public
- `/` -- LearningHub (Beeducation)
- `/chat` -- Chat with Mochi
- `/learning/bee-basics`, `/learning/garden-basics`
- `/auth`
- Landing, Onboarding, Language flows (unchanged)

### New files & changes

**1. `src/pages/Admin.tsx`** (new)
- A tabbed admin layout with tabs: **Control Panel**, **Production**, **Technical Details**, **Brand Book**, **Design System**, **Privacy**
- Imports existing components: `MasterControlPanel`, `ProductionDashboard`, `TechnicalDetails` content (inline or as component), `ConsentSettings`
- Contains the two new sections below
- Gated: checks `user_roles` table for admin role via `has_role` Supabase function. If not admin, shows "Access Denied" message. Falls back gracefully for now (shows content if no auth, since roles may not be seeded yet -- we'll add a simple localStorage admin flag as a UI convenience, but the real check is server-side via RLS).

**2. Brand Book tab inside Admin** (new component `src/components/admin/BrandBook.tsx`)
- Displays the MochiBee brand identity:
  - **Logo usage**: Shows the Mochi bee logo, La Pipa logo, correct sizing/spacing
  - **Color palette**: Primary (yellow/amber), secondary (green), accent (blue), with hex codes and Tailwind class names
  - **Typography**: Font stack, heading sizes, body text specs
  - **Tone of voice**: Friendly, educational, bee-themed, bilingual guidelines
  - **Mascot guidelines**: Mochi character usage rules
- All static content, styled with existing glass-card and design tokens

**3. Design System tab inside Admin** (new component `src/components/admin/DesignSystem.tsx`)
- Live component showcase rendered with actual UI components:
  - **Buttons**: All variants (default, outline, ghost, destructive) rendered live
  - **Cards**: glass-card, standard card, gradient cards
  - **Badges**: All color variants used in the app
  - **Typography scale**: h1-h6, body, caption rendered live
  - **Animations**: bouncy-enter, bee-flutter, float-up demos
  - **Spacing & Layout**: padding/margin scale reference
  - **Form elements**: Input, Select, Checkbox, Switch samples
- Each section has a title and the Tailwind class name for easy copy

**4. Route changes (`src/App.tsx`)**
- Add `/admin/*` route pointing to the new Admin page (lazy-loaded)
- Remove `/dashboard`, `/production`, `/technical-details` from public routes
- Keep all other routes unchanged

**5. Navigation updates**
- **`src/components/AppHeader.tsx`**: Remove any dashboard nav links from the public header (none currently visible in desktop nav, just Beeducation + Chat -- keep as-is)
- **`src/components/HamburgerMenu.tsx`**: Replace "Technical Details" link with "Admin Panel" link (only shown if user is authenticated). Remove "System Help" button from public menu, move to admin. Keep "About Mochi" and "Lapipa.io" link.

**6. `src/components/admin/AdminGuard.tsx`** (new)
- Wrapper component that checks if user is authenticated
- For now, any authenticated user can access admin (role enforcement via RLS on data queries)
- Shows a sign-in prompt for unauthenticated users

### Safety
- No public pages are modified in content, only routes are reorganized
- All existing components are reused, not rewritten
- Lazy loading for the admin section keeps bundle size unchanged for public users
- No database changes needed -- existing `user_roles` table and `has_role` function already exist
